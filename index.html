DVWA(4)-XSS

发表于 2017-12-10

Javascript的出现，使得恶意代码能够在用户的浏览器端执行，取得敏感信息。为了防止这种现象，实施了同源策略。与之相应的，如今的浏览器以及服务端也实行了访问控制。

而XSS(cross site script)能将恶意代码(通常是前端的)注入到可信的站点，当用户访问的时候，由于浏览器认为该代码是可信的，所以会在用户的浏览器中执行它。恶意代码也得以顺利访问用户的cookie，session或其他敏感信息。网站的所有者无法在 html 源码中看到恶意脚本，也无法在 html response 中看到。

XSS-DOM

xss攻击主要可以分为 Reflected XSS ，Stored XSS，不过还有一种相对不太著名的攻击方式： DOM based XSS

这种攻击方式的环境都在客户端，与服务端基本没有什么关系。也就是说，虽然服务器端传回的是正常页面，但是由于用户端存在恶意代码，使得页面在浏览器中render之后变得不正常。

low-level

界面

low-level下，依然是没有任何防护。在参数中注入怎样的js代码，这段js代码都会被原原本本地write到页面中。

其实服务器端发给客户端的页面没有发生任何变化。在客户端，js代码运行后页面才被影响。

high-level

impossible-level

DVWA 中没有给出服务端的防护代码，因为服务器端对于阻止这种攻击无能为力，问题出在前端。
主要还是得尽量避免客户端处对网页文档的修改，而采用配合服务端处使用动态页面技术来更改页面。

不过现代浏览器在前端安全中也扮演着重要的角色。令我印象深刻的就是Chrome 的 XSS Auditor 能够自动屏蔽疑似 XSS 注入的 Payload。

应对措施

可以设置 cookie 为 http_only，该属性能使得 cookie 无法被 js 调用。这样一来即使用户受到了 XSS 攻击，攻击者也无法通过恶意 js 代码得到cookie信息。
比如 PHP可以设置配置文件php.ini中的 session.cookie_Httponly
其他语言设置http_only的方法可参考链接

XSS-Reflect

这种攻击方式中，由于恶意代码并不存放在服务端，因此要触发攻击需要一些社工(与 CSRF 有些相似)。一般以电子邮件链接等方式触发攻击。当用户被诱导点击恶意链接时，该链接将发送一段被构造过的请求，或重定向用户访问有漏洞的网站，然后利用漏洞运行影响用户浏览器的恶意代码。

impossible-level

加入了 Token，并使用 htmlspecialchars() 函数过滤输入。

该函数一般用来将HTML中的一些保留字符(比如 <,>,",',& 等字符)转换成HTML Entities，防止出现混淆。

如果我们输入的值是放在html标签内的，那么我们可能还有机会能通过事件函数等方式运行js代码。但是如果是放在标签外部，那么能成功注入的机会就很渺茫了。

Reflect XSS 是 XSS 中分类最多的攻击方式，这里出现的情况很有限，更多的还得自己去发现。

XSS-Stored

一般在诸如论坛、评论区等地方，都会将用户的输入存放到服务器的数据库中，当别的用户浏览的时候从数据库中取出发送。那么如果存放到服务端的数据库中的为恶意代码，那么浏览这些内容的用户就会受到影响。(如今HTML5中好像有个 web storage 可以在客户端中存储数据，从而不需要每次都要去服务端取数据。这似乎也同时方便了 xss-stored 漏洞，使恶意代码可以在客户端长期存储。)

DVWA(3)-CSRF

发表于 2017-12-03

csrf 全名 cross site request forgery(跨站请求伪造)。

该攻击的核心在于 “服务器信任用户的浏览器所发出的请求就是用户想要的请求”。利用的思想是 想办法取得服务器对该用户的信任(cookie,session等)，并让浏览器发送恶意请求(加载图片，js脚本等)。

许多网站对于用户的认证信息都保存在了浏览器中，比如我们访问许多网站经常都可以跳过输入账号密码的步骤。这使得服务器端对于敏感操作如果不加有效的用户认证的话，容易遭受 csrf 攻击。

DVWA(2)-command injection

发表于 2017-11-22

该漏洞允许用户在服务器上执行任意命令。

该命令与 code injection 的区别在于：
code injection 与注入的语言有关，比如 服务器上执行的语言为 php，那么可利用漏洞执行 php 代码。
如PHP中的 require，eval() 等，都容易引发 code injection 漏洞。

而 command injection 常常发生于 服务器对输入 不加有效过滤而传给 system shell 执行，从而被恶意利用。如果被利用的程序权限较高，那么后果将是灾难性的。
