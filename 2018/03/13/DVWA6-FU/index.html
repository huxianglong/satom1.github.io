DVWA(6)-File Uploads

发表于 2018-03-13

网站启用文件上传的功能是一个有很大风险的行为。如果恶意用户成功上传了含有恶意代码的文件，那么接下来只需要想办法执行该文件–直接访问，或者用文件包含(如果运气实在不错，可以利用eval())。
因此，必须仔细过滤用户上传的文件。

low-level

  • 上传的方式为POST。
    • 全局变量$_FILES 包含上传文件的信息。
    • upload为上传表单中指定的字段
    • $_FILES[ 'uploaded' ][ 'name' ]为文件在客户端的原名称
    • $_FILES[ 'uploaded' ][ 'tmp_name' ]为文件被上传后在服务端储存的临时文件名。
  • move_upload_file()函数先检查文件以确保其为有效的上传文件(是否通过PHP POST方式上传)，若检查通过则将文件转移到指定目录中。

这里没有对上传文件进行任何检查，而且上传的路径也返回给了用户。

那么我们直接访问 http://172.16.65.145/DVWA/hackable/uploads/test.php， 可以执行此文件。

high-level

主要作了以下改动：

  1. 函数 getimagesize() 会检查文件的头部，当其不是一个正常图片文件的头部时，会返回False。若为正常图片，则返回图片的尺寸等数据。
    这样我们需要想办法将代码嵌入到正常图片内。可用GIMP等工具进行编辑。这里用Python作一下演示：

    这样，生成的 newimage.png 就内嵌了 code.php 文件中的php代码。

  2. 函数strrpos()的含义为

    int strrpos ( string $haystack , string $needle [, int $offset = 0 ] )
    Find the numeric position of the last occurrence of needle in the haystack string.

    若上传的文件有多个后缀，则这里会检查最后的后缀。 而刚好 Apache 在遇到双后缀的文件时一般也是以后者为准。

也就是说，用 直接访问文件来执行其中的代码 这种方法已经不可行了。 接下来可以考虑结合其他漏洞，比如 File Inclusion。

在前面的 File Inclusion篇中，可用high-level 下介绍的 file:// 协议，将我们前面使用Python嵌入了PHP代码的图片上传，并进行包含。

impossible-level

  • imagecreatefromjpeg()
    根据参数给的图片，创建一张新的图片，函数返回一个图像标识符。
    新的图片中没有 metadata 等额外数据，意味着上传的图片会被“消毒”。 imagecreatefrompng同理。
  • imagejpeg()
    以一个图像标识符为参数，输出图片到浏览器或文件中。[2]表示输出的目的地。[3]表示图片质量。imagepng同理。
  • imagedestroy()
    清除内存中与参数有关的图像标识符。
  • rename()
    重命名文件。必要情况下可以移动文件至其他路径。
  • unlink()
    删除文件。

可以看到这些改变：

  1. $target_file不再是上传的文件名，而是用 md5() 再加上时间戳 使服务器上的文件名难以得知。
    这样一来，如果没有文件读取的漏洞的话，已经很难利用该图片了(然而实验中会一直返回上传的文件名)。
  2. 以上还不是最主要的防御措施。imagecreatefromjpeg() -- imagejpeg() -- imagedestroy() -- rename() -- unlink() 这一系列函数才真正让嵌入代码到图片完全失去意义。服务器创建一个不带任何杂质的图片副本，再将客户端上传的图片完全从服务器中删去。

总结：

  • 服务器端识别文件类型时需要注意的几点：
    1. 如果服务器仅支持一种脚本语言，则可以考虑根据文件后缀名过滤。对于支持多种语言的服务器来说，强烈建议使用白名单过滤。
    2. 检查不同服务器对应的设置，当出现双后缀名，或其他特殊分隔符时如何处理。(如file.php.jpg等)
    3. 推荐使用函数 basename()、realpath()
    4. 更多的，可以参考 impossible-level
  • 防止用户上传的文件被包含
    将应被包含的文件与用户上传的文件放于不同路径下，包含前先检查路径。
